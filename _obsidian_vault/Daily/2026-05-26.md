## HermesGuildCore Guild UI Runtime Handoff

### Context

User corrected the earlier progress estimate: the Guild app should not be treated as "80% done" if measured against the intended architecture.

Intended architecture:

- Hermes/Guild Master splits a product into bounded modules.
- Each worker owns one module based on rank/skill/provider capability.
- Workers publish artifacts to the blackboard.
- A join-review / meeting task reads artifacts, detects mismatches, and creates bounded fix tasks for the responsible module/worker.
- Fix tasks run, publish new artifacts, and the meeting repeats until final product or stop condition.
- UI should make this visible; hidden background work without logs is unacceptable.

Current reality before today's fixes:

- UI could create a quest, open worker terminals, call provider adapters, publish artifacts, and show task state.
- Worker tasks were too generic (`required_skill=general`), so workers could claim the wrong module.
- Review/finalize mostly waited for all build tasks to be done; if one failed, the board appeared stuck.
- OpenCode worker calls could fail with Windows `The command line is too long` because the adapter passed the whole prompt through argv.
- Dashboard server was launched hidden without obvious stdout/stderr logs.

### Architecture Clarification For Afternoon

Decision:

- Providers do not need to "enter Guild" or understand the whole Guild runtime.
- The correct contract is: Guild creates bounded tasks, worker wrapper claims tasks, provider performs only the assigned task packet, worker wrapper validates output, then publishes artifact back to Guild.
- Provider sandbox/network quirks are environment problems, not the core architecture criterion.

Current direction is right:

- Dashboard/user gives task to Hermes/Guild.
- Guild/Hermes creates task contracts and allowed scope.
- Workers claim from blackboard.
- Providers are called through adapters.
- Output validation prevents bad provider output from being marked done.

Important correction:

- Current `/api/hermes/finalize` works as a pragmatic bridge, but it is still API-owned.
- Cleaner Guild model: finalization/meeting should become a normal `join_review` or `final_review` task that is claimed, executed, validated, and published through the same worker-wrapper/artifact path.
- Afternoon priority should be this correction before adding more scheduler complexity, unless visible smoke exposes a more urgent bug.

### Book Insight

We are not forcing Guild to follow the book as doctrine. We only record useful ideas and keep the ones that fit the system.

From the blackboard / multi-agent flow analysis in the book-like material:

- The useful pattern is not "many agents talking", but "shared state + specialists + orchestrator".
- Specialists should wake only when the shared state makes them relevant, not by constant polling.
- Triggering should be event-driven and bounded, otherwise the system turns into noisy scanning.
- Concurrency can work if shared state is narrow and contention is controlled at the state layer, not by letting agents free-run.
- Data and control should be separated: agents reason over state, but the runtime owns scheduling and termination.

What this means for Guild:

- Keep blackboard/task/artifact as the center of gravity.
- Keep workers narrow and role-bound instead of general-purpose.
- Prefer runtime-enforced triggers and validations over prompt-only instructions.
- Do not let Hermes/worker behavior rely on "hope the model follows the rule"; encode the rule in launcher/wrapper/runtime checks.

Main warning from the book insight:

- If the system only has a shared board but no disciplined triggering/termination, it will drift into scan-noise and ambiguous completion.
- So the next layer should be more enforcement and audit, not more agent freedom.
